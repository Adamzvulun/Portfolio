TL;DR

An external review graded the site B+ and flagged 10 issues. We audited every claim against the real codebase before planning: 5 confirmed, 3 overstated, 2 simply wrong — including the single "critical blocker," which turned out not to block anything (the production build passes; verified).

The resulting plan has 5 phases, roughly 4–6 hours total, ordered by user impact: a 5-minute dependency fix, then image loading speed (LCP), then keyboard & screen-reader accessibility, then SEO (sitemap, robots, structured data), then config cleanup.

Overstated

"Missing sharp — build will fail in CI/CD" (rated Critical)

The build passes. We ran npm run build on a clean install: exit 0, all 9 routes prerendered. sharp@0.34.5 arrives transitively through next@16.2.7 itself. The real (smaller) risk: transitive hoisting isn't a contract — a Next upgrade or pnpm's strict mode would break it. Worth one line in package.json, not a fire drill.

Wrong fix

"Add priority prop to first images"

The problem is real (no image is prioritized) but the prescribed fix uses a deprecated API: Next 16 deprecated priority in favor of preload, and its docs recommend fetchPriority="high" / loading="eager" when several images could be the LCP element — exactly our two-column gallery. Following the review verbatim would have added deprecation warnings.

Overstated

"outline-none removes all focus indicators"

The same inputs carry focus:border-neutral-900, so a visible indicator exists — the border darkens on focus. It's a weak affordance though; we'll upgrade to a proper focus-visible ring across all interactive elements.

Verify first

"Lightbox lacks a focus trap"

Speculation ("focus likely stays on the trigger"). The lightbox library (yet-another-react-lightbox) ships its own focus management. Plan: a 5-minute keyboard test before scheduling any work — probably a no-op.

Confirmed

No sitemap.xml or robots.txt

Nothing under app/. Next 16's app/sitemap.ts and app/robots.ts file conventions make both auto-generated routes. Scheduled in Phase 4.

Confirmed

No JSON-LD structured data

Nothing emits application/ld+json. A Person schema in the root layout is the right starting point for a photographer's portfolio. Scheduled in Phase 4 — with one constraint the review missed (see new finding C).

Confirmed

Form errors don't auto-clear

ContactForm.tsx clears the error on resubmit but not when the user edits a field. Small UX polish, scheduled in Phase 3.

Confirmed, gated

Redundant outputFileTracingExcludes entry

Two keys with identical globs in next.config.ts. But this exact config once fixed a 559 MB serverless function that blew Vercel's 300 MB limit — it's load-bearing. Deduplicate only with a verified preview deploy. Phase 5.

Wrong

"Nav's links array recreated on every render"

The array is already module-level (Nav.tsx:6), outside the component. Already done; no action.

Not found

"Misleading lightboxIndex comment in Gallery.tsx"

No such comment exists in the current file; the comments that are there (layout modes, mobile column) are accurate. No action.

1b · What the review missed

Three things surfaced during the audit that the review didn't catch.

New — A

The fade-in animation delays LCP more than the missing preload does

Every tile renders at opacity-0 until its onLoad fires. Browsers exclude invisible elements from LCP candidacy, so the metric is pushed past the 700ms fade. Fix together with the preload work in Phase 2: first-viewport tiles skip (or shorten) the fade.

New — B

Form status changes are invisible to screen readers

"Thank you" / error messages swap in visually with no aria-live / role="status", so assistive tech never announces them. Cheap fix, bundled into Phase 3.

New — C

A constraint to protect: the site intentionally has no meta description

Documented in app/layout.tsx — any description tag resurfaces as a subtitle on WhatsApp share cards. SEO work (Phase 4) must not "helpfully" add one. Sitemap, robots and JSON-LD are all safe; they emit no meta tags.

Where each phase lands. Top-left is where you want to live: high impact, low effort.

3 · The Phases

Each phase is independently shippable — stop after any one of them and the site is strictly better. One PR per phase keeps deploys verifiable.

Phase 1

Dependency correctness

~5 min

The cheapest insurance in the repo — protects the build from a future Next upgrade or package-manager change silently dropping sharp.

  • Add sharp to dependencies (npm install sharp) — it's imported by lib/photos.ts and scripts/resize.mjs.
Done when: npm ls sharp shows a direct dependency and npm run build is still green.
Phase 2

Perceived performance (LCP)

~1–2 h

The homepage is the storefront. The first paint of the first photos is the whole first impression — and right now they're lazy-loaded and hidden until their fade-in starts.

  • Give the first ~2 tiles per column fetchPriority="high" + loading="eager" — the Next 16 way (not the deprecated priority prop the review suggested).
  • Exempt first-viewport tiles from the opacity-0 → onLoad fade so the LCP element paints visible (new finding A).
  • Sanity-check the sizes attribute against the real rendered column width — verify, don't assume.
Done when: mobile Lighthouse LCP on / measurably improves; no blank flash on first-viewport photos; zero deprecation warnings.
Phase 3

Accessibility

~1–2 h

Keyboard and screen-reader users currently get a weak focus indicator on the form and no announcements when it succeeds or fails.

  • Visible focus-visible ring on form fields (replacing outline-none), gallery tiles, nav links, Send and BackToTop buttons.
  • Announce form status: role="status" / aria-live="polite" on the success panel and error message (new finding B).
  • Clear the form error as soon as the user edits a field.
  • Keyboard-test the lightbox (Enter opens, Tab stays inside, Esc closes, focus returns). Only schedule fixes if the test actually fails.
Done when: tabbing through the whole site always shows where you are; a screen reader announces form results; lightbox keyboard behavior is documented pass/fail.
Phase 4

SEO

~1–2 h

Make the site legible to crawlers — without breaking the deliberate no-meta-description WhatsApp workaround (new finding C).

  • app/robots.ts — allow all, point at the sitemap.
  • app/sitemap.ts — the five static routes on the adamzvulun.com base.
  • JSON-LD Person schema (photographer) in the root layout, rendered as an inline script with < escaped, per the Next 16 docs. Per-gallery ImageGallery schema as an optional follow-up.
Done when: /robots.txt and /sitemap.xml serve from a local build; JSON-LD passes Google's Rich Results test; the WhatsApp share card is unchanged.
Phase 5

Cleanup (verify-gated)

~30 min

Low stakes, but touches a config that once kept the Vercel deploy under the 300 MB function limit — so it ships only with a verified preview deploy.

  • Deduplicate the two identical outputFileTracingExcludes keys in next.config.ts; confirm function size on the preview deploy before merge.
  • Update PLAN.md per its own "keep this file fresh" convention.
Done when: Vercel preview deploy succeeds with the function comfortably under the limit.

5 · Explicitly Not Doing

Items from the review that audit showed to be moot — or fixes that would actively break something intentional.

  • Hoisting the Nav links array — already module-level; the review misread the file.
  • Fixing the "misleading lightboxIndex comment" — no such comment exists.
  • Adding a meta description — intentionally omitted; it would resurface on WhatsApp share cards (documented in layout.tsx).
  • Pre-emptive lightbox focus-trap rework — the library likely handles it; test first, work only on failure.
